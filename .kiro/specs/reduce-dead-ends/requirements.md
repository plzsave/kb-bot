# Requirements Document

## Introduction
kb-bot の北極星は「非エンジニアが独力で疑問を解決する」こと。これが崩れる典型が、ドキュメントに無い質問で bot が実コードを見に行かず「知識ベースに見つかりませんでした」で行き止まるケースである。`buildSystem` は GitHub 有効時に「コード優先」を促すが、"見つからない" を宣言する前にコードを確認することを必須化していないため、実装・挙動・仕様・コスト系の質問でもコードを見ずに諦めうる。

本仕様はこの 1 点に絞って対処する: **GitHub 有効時、"見つからない" と述べる前に実コードを確認することを必須化する**。これは今回報告された「コードを見に行かない」劣化への直接の効き目があり、プロンプトの決定的変更で実現でき、曖昧さがない。

「未発見/諦め回答をキャッシュしない」対処は、頑健にやるには retrieval の関連性スコア（`dropWeakHits` が無関係ヒットを残すため FTS 件数では実質空振りを判定できない）に依存する。よって**キャッシュ対処はスコア化エスカレーション（A 軸）follow-on スペックへ移し、本仕様では扱わない**。基本モデル底上げ・eval ケース追加も対象外。詳細は brief.md 参照。

## Boundary Context
- **In scope**:
  - GitHub 有効時、未発見宣言前に実コードを確認させる本番プロンプトの挙動（`buildSystem`）
  - 上記の決定的な回帰テスト（プロンプト出力の検証）
- **Out of scope**:
  - 未発見/諦め回答の非キャッシュ化 ＝ A経路スコア化 follow-on スペックへ移管（関連性スコアが入って初めて決定的に判定できるため）
  - A経路のスコア化エスカレーション（実質空振り検知・昇格トリガ・retrieval 関連性しきい値）
  - 基本モデル底上げ（`.env`・コード外）
  - eval ハーネスの判定ロジックや新規 eval ケース（`eval-*` 各スペックが所有。効果測定は既存 B′/D で行う）
- **Adjacent expectations**:
  - 未発見時の「次の一歩」提示（#31 / eval-next-step）は既存挙動として保持し、本仕様の "見つからない" 宣言はこれと共存する
  - docs 優先のルーティング（手順・ルール・用語は docs、実装・挙動・仕様はコード）は維持する
  - GitHub 無効時は参照すべきコードが無いため、本仕様の必須化は適用されない

## Requirements

### Requirement 1: 未発見宣言前の実コード確認（GitHub 有効時）
**Objective:** As a 非エンジニアの利用者, I want 仕組み・挙動・仕様・コストの質問で bot が諦める前に実コードを確認してほしい, so that ドキュメントに無くてもコードから答えを得られ、行き止まらない

#### Acceptance Criteria
1. While GitHub 参照が有効, when 実装・挙動・仕様・コストに関する質問を受け初期コンテキスト（docs）だけでは答えられない, the kb-bot shall 「見つからない」と述べる前に実際のコードを確認する（コード参照ツールでリポジトリを調べる）
2. While GitHub 参照が有効, when docs にもコードにも事実が見つからない, the kb-bot shall 初めて「見つからない」旨を述べ、あわせて次の一歩（#31 の挙動）を提示する
3. When docs（初期コンテキストまたは知識検索）だけで十分に答えられる, the kb-bot shall 不要なコード確認を強制されず、docs 優先の既存ルーティングを保つ
4. While GitHub 参照が無効, the kb-bot shall コード確認を要求されず、従来どおり docs の範囲で「見つからない」旨と次の一歩を返す
5. The プロンプト変更 shall [Safety]・[Output style]・#31 の次の一歩・出力言語自動判別の各既存文言/挙動を弱めない

### Requirement 2: 既存資産の非回帰
**Objective:** As a 保守者, I want 本変更が既存の挙動・テスト・別スペックの所有範囲を壊さないことを保証したい, so that 変更を安全に取り込める

#### Acceptance Criteria
1. When `bun run typecheck` を実行, the プロジェクト shall 型エラーなく完了する
2. When 既存テスト（`bun test`）を実行, the プロジェクト shall 既存テストを無改修で維持する（壊さない）
3. The 変更 shall 回答キャッシュの保存/照会ロジック（`src/cache.ts` および `core.ts` の保存条件）を変更しない（キャッシュ対処は follow-on スペックが所有）
4. The 変更 shall スコア化エスカレーション（`startHard`/retrieval 関連性）と基本モデル設定を変更しない（別 follow-on が所有）
5. The 変更 shall `src/chat/core.ts` の `buildSystem` 以外の本番コードを改変しない
