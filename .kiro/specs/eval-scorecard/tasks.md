# Implementation Plan

- [ ] 1. 基盤: エントリ分離と型スキャフォールド
- [x] 1.1 実行をエントリ限定にし、軸・ゲートの型土台を用意する
  - トップレベルで走っている読込・LLM/GitHub 初期化・評価ループ・終了処理を一つのエントリ関数にまとめ、直接実行時のみ動くガード配下に閉じる
  - 評価軸の許容集合（A/B/C/D/safety）と、未検証の生ケース形状（軸・ゲートは緩い型）／検証済みケース形状（軸は有効値・ゲートは真偽値に確定）を定義する
  - 既存の採点ロジックと SKIP 判定は一切変更しない（呼び出すだけ）
  - 完了状態: モジュールを import しても LLM/GitHub 呼び出しが走らず、`bun run scripts/kb-eval.ts` の従来挙動が保たれる
  - _Requirements: 4.2, 4.4_

- [ ] 2. コア: 純粋関数（検証・集計・合否・整形）
- [x] 2.1 ケース検証で不正な評価軸を弾く
  - 生ケース列を受け、各ケースの評価軸が許容集合に含まれるか、ゲート指定が真偽値かを検証し、不正項目のエラー文を集める（副作用なし）
  - 検証は許容集合への narrow 前に行い、不正値が黙って集計へ流れ込まないようにする
  - 完了状態: 不正軸（例 "X"）を含むと非空のエラー列を返し、有効軸・無タグのみなら空列を返す
  - _Requirements: 1.4_
  - _Boundary: CaseValidation_

- [x] 2.2 評価済みケースからスコアカードを集計する
  - 評価済み（非 SKIP）のケースのみを対象に、軸別 pass/total、ゲートの失敗一覧と母数、総合（pass・評価済み件数・SKIP 件数）を算出する
  - 無タグは軸別に含めず総合のみに数え、SKIP は軸別・ゲート・評価済み件数のいずれにも数えない
  - 評価軸とゲートは直交として扱い、両方を持つケースは軸別 tally とゲート母数の双方に計上する
  - 完了状態: 軸・ゲート・無タグ・SKIP を混在させた結果列から、SKIP を除外した軸別 tally とゲート失敗一覧・総合内訳が得られる
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 3.2, 5.2, 5.3_
  - _Boundary: EvalScorecard_

- [x] 2.3 全体合否の判定とスコアカードの整形を実装する
  - スコアカードから全体合否を「評価済み全 PASS かつ ゲート失敗なし」で導く純粋判定を実装する
  - スコアカードを末尾表示用の文字列（軸別行＋ゲート行＋総合行）に整形し、総合行は評価済み基準で PASS 数・SKIP 数を明示する（既存の総合 PASS 数を保持）
  - 完了状態: ゲート失敗ありで合否判定が不合格になり、整形出力に軸別行・ゲート行・総合 PASS 数が含まれる
  - _Requirements: 1.2, 2.2, 2.4, 3.1, 3.3_
  - _Boundary: EvalScorecard_

- [ ] 3. 統合: 評価ループへの結線と終了コード反映
- [x] 3.1 ループ結果をスコアカードへ写像し、終了コードに反映する
  - エントリ関数の冒頭で検証を呼び、不正があれば内容を出力して非ゼロ終了する
  - 各ケースの結果（PASS/FAIL/SKIP/ERROR）を集計入力へ写像し、ゲートの失敗は逐次行でスコア軸の失敗と区別できる印を付けて出力する
  - ループ後に集計・整形・合否判定を呼び、スコアカードを末尾に追加表示し、全体合否に応じた終了コードで終了する（GitHub 未設定時の SKIP 挙動は維持）
  - 完了状態: 実行末尾にスコアカードが表示され、ゲート失敗が区別表示され、SKIP のみで他が PASS なら終了コード 0 になる
  - _Requirements: 1.4, 2.3, 3.1, 5.1, 5.2_

## Implementation Notes
- 純粋関数（`validateCases`/`buildScorecard`/`overallPassed`/`formatScorecard`）と型（`Axis`/`AXES`/`RawCase`/`Case`/`CaseStatus`/`CaseResult`/`AxisTally`/`Scorecard`）は `scripts/kb-eval.ts` で `export` 済み。3.1 の main 結線・4.1 のテストはこれらを再利用し再定義しないこと。
- 2.3 で確定した `formatScorecard` の出力ラベル（`=== スコアカード ===` / `軸 X: p/t PASS` / `ゲート: FAIL n 件（…）` / `総合 p/e PASS, s SKIP`）は design の Revalidation Triggers 対象。3.1 はこのフォーマットをそのまま末尾出力に使い、勝手に変えないこと。逐次行のゲート区別は別途 `FAIL*`（gate 印）で行う。
